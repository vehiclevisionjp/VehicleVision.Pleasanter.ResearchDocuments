# 複数インスタンス間マスターデータ同期設計

複数のプリザンターインスタンス間で特定のサイト（マスターデータ）を同期させる外部アプリケーションの設計調査。プリザンター本体には手を加えず、RDBMS を直接操作することで同期を実現する。対等型・親子型のトポロジに対応し、項目単位・方向単位・レコード単位の同期制御を備えた仕組みを検討する。Results / Issues に加え、Wikis および添付ファイル（Binaries テーブル・RDS 保管時）も同期対象とする。

## 調査目的

- 複数のプリザンターインスタンス間でマスターデータ（選択肢テーブル等）を同期させる仕組みを設計する
- **プリザンター本体を改修せず、外部アプリケーションとして実装する**（既存環境のバージョンアップが困難なため）
- **API を使用せず RDBMS を直接操作する**ことで、高速かつ確実な同期を実現する
- 対応 RDBMS は **SQL Server・MySQL・PostgreSQL** の 3 種とする
- 対等型（Peer-to-Peer）と親子型（Hub-Spoke）の 2 つのトポロジパターンを整理する
- 項目単位・方向単位・インスタンス単位・**レコード単位**で同期範囲を制御できる仕組みを検討する
- **Wikis テーブル**（Wiki ページ）も同期対象とする
- **添付ファイル（Binaries テーブル）** も同期対象とする（RDS 保管時に限定）
- インスタンス間で異なる各種 ID（SiteId・ResultId・UserId 等）の**マッピング機構**を備える
- できるだけ**リアルタイムに近い同期**を実現する

## 設計方針

| 項目                   | 方針                                                       |
| ---------------------- | ---------------------------------------------------------- |
| 実装形態               | プリザンターとは独立した外部アプリケーション               |
| データアクセス         | RDBMS への直接 SQL 操作（API 不使用）                      |
| 対応 RDBMS             | SQL Server・MySQL・PostgreSQL                              |
| 同期タイミング         | リアルタイム（変更検出ベース）を基本とし、バッチ補完を併用 |
| プリザンター本体の改修 | 不要（既存環境をそのまま利用可能）                         |

### API 不使用の理由

| 観点           | API 方式の課題                               | DB 直接方式の利点                                     |
| -------------- | -------------------------------------------- | ----------------------------------------------------- |
| バージョン依存 | API 仕様がバージョンにより変わる可能性がある | テーブル構造は安定しており変更頻度が低い              |
| パフォーマンス | HTTP オーバーヘッド・認証処理が発生する      | DB 接続で直接操作するため高速                         |
| 一括処理       | BulkUpsert の制約（タイムアウト等）          | SQL の INSERT/UPDATE で大量データを効率的に処理できる |
| 競合制御       | Upsert API に DB レベルのロックがない        | トランザクションと行ロックで競合を確実に防止できる    |
| 削除検出       | API では削除済みレコードを取得できない       | `_deleted` テーブルを直接参照して削除を検出できる     |
| 導入容易性     | 各インスタンスの API キー管理が必要          | DB 接続情報のみで運用可能                             |

## プリザンターのデータベース構造

外部同期アプリケーションが操作する主要テーブルの構造を整理する。

### テーブル構成

プリザンターのデータは以下のテーブル群で管理される。

```mermaid

    Items ||--o| Issues : "ReferenceType='Issues'"

    Sites {
        int SiteId PK

    Results {
        bigint ResultId PK

    }
    Issues {
        bigint IssueId PK
```

### テーブルバリアント（3 テーブル構成）

各ビジネステーブルは 3 つのバリアントで構成される。

| テーブル          | 用途                   | 同期での扱い           |
| ----------------- | ---------------------- | ---------------------- |
| `Results`         | アクティブなレコード   | 同期の主対象           |
| `Results_deleted` | 論理削除されたレコード | 削除同期の検出に使用   |
| `Results_history` | 更新履歴               | 同期対象外（参考情報） |

同様に Wikis テーブルも 3 バリアント構成を持つ。

| テーブル        | 用途                   | 同期での扱い         |
| --------------- | ---------------------- | -------------------- |
| `Wikis`         | アクティブなレコード   | 同期の主対象         |
| `Wikis_deleted` | 論理削除されたレコード | 削除同期の検出に使用 |
| `Wikis_history` | 更新履歴               | 同期対象外           |

Binaries テーブルは `_history` バリアントのみ存在し、`_deleted` バリアントは持たない。

| テーブル           | 用途                 | 同期での扱い                   |
| ------------------ | -------------------- | ------------------------------ |
| `Binaries`         | アクティブなレコード | 同期の主対象（RDS 保管時のみ） |
| `Binaries_history` | 更新履歴             | 同期対象外                     |

### Items テーブルとの関係

Results / Issues テーブルの各レコードには、対応する Items テーブルのレコードが存在する。

| Items カラム    | 説明                                                     |
| --------------- | -------------------------------------------------------- |
| `ReferenceId`   | Results.ResultId / Issues.IssueId / Wikis.WikiId に対応  |
| `ReferenceType` | `"Results"` / `"Issues"` / `"Wikis"`                     |
| `SiteId`        | レコードが属するサイトの ID                              |
| `Title`         | レコードタイトル（Results/Issues/Wikis の Title と同値） |
| `UpdatedTime`   | 最終更新日時                                             |

> **重要**: レコードを同期する際は、Results/Issues/Wikis テーブルと Items テーブルの
> **両方を整合的に更新する**必要がある。
> 添付ファイルがあるレコードでは Binaries テーブルも合わせて同期する。

### 同期対象テーブルの判定

サイトがどのテーブルにデータを持つかは `Sites.ReferenceType` で判定する。

```sql
-- 同期対象サイトのテーブル種別を取得
SELECT "SiteId", "ReferenceType"
FROM "Sites"
WHERE "SiteId" IN (対象SiteId一覧)
```

| ReferenceType | データテーブル | 主キー     |
| ------------- | -------------- | ---------- |
| `Results`     | Results        | `ResultId` |
| `Issues`      | Issues         | `IssueId`  |
| `Wikis`       | Wikis          | `WikiId`   |

### 同期キーとなるカラム

同期元と同期先でレコードを対応付けるためのキーカラムを設計する。

| カラム種別                    | 説明                                                      |
| ----------------------------- | --------------------------------------------------------- |
| `ResultId`/`IssueId`/`WikiId` | インスタンスごとに自動採番されるため同期キーに使用不可    |
| `ClassA`〜`ClassZ`            | 業務キー（従業員番号・商品コード等）を格納するカラム      |
| 複合キー                      | 複数の Class カラムの組み合わせで一意性を担保             |
| `Title`（Wikis 用）           | Wiki はタイトルを同期キーとして使用可能（サイト内で一意） |

## 同期トポロジの整理

### パターン 1: 親子型（Hub-Spoke）

1 つの親インスタンスが正本を持ち、複数の子インスタンスにデータを配信する構成。

```mermaid
graph TD
    Parent["親インスタンス（正本）"]
    ChildA["子インスタンス A"]
    ChildB["子インスタンス B"]
    ChildC["子インスタンス C"]

    Parent -->|同期| ChildA
    Parent -->|同期| ChildB
    Parent -->|同期| ChildC
    ChildA -.->|差戻し| Parent
    ChildB -.->|差戻し| Parent
```

| 特性       | 内容                                           |
| ---------- | ---------------------------------------------- |
| 正本の所在 | 親インスタンスが唯一の正本                     |
| 同期方向   | 基本は親→子の一方向。子→親は制限付き（差戻し） |
| 競合リスク | 低い（正本が一意に決まるため）                 |
| 適用例     | 本社→支社、管理サーバ→現場端末                 |

### パターン 2: 対等型（Peer-to-Peer）

全インスタンスが同等の権限を持ち、どこで変更しても他のインスタンスに反映される構成。

```mermaid
graph LR
    PeerA["インスタンス A"]
    PeerB["インスタンス B"]
    PeerC["インスタンス C"]

    PeerA <-->|同期| PeerB
    PeerB <-->|同期| PeerC
    PeerC <-->|同期| PeerA
```

| 特性       | 内容                                       |
| ---------- | ------------------------------------------ |
| 正本の所在 | 全インスタンスが対等（正本なし）           |
| 同期方向   | 双方向                                     |
| 競合リスク | 高い（同一レコードの同時編集が発生し得る） |
| 適用例     | 拠点間で同等の運用を行うケース             |

### トポロジの選定指針

| 判断基準                 | 親子型推奨   | 対等型推奨         |
| ------------------------ | ------------ | ------------------ |
| マスターデータの管理拠点 | 1 拠点に集約 | 複数拠点で分散管理 |
| 競合発生時の解決コスト   | 低い         | 高い               |
| 実装の複雑度             | 低い         | 高い               |
| 運用の柔軟性             | 限定的       | 高い               |

## 同期制御の設計

### 制御の粒度

同期対象を細かく制御するため、以下の 4 レベルの制御粒度を設ける。

| 制御レベル         | 説明                                         | 設定例                                               |
| ------------------ | -------------------------------------------- | ---------------------------------------------------- |
| サイト単位         | 同期対象のサイト（テーブル）を指定する       | サイト A は同期対象、サイト B は対象外               |
| レコード単位       | 同期キー値に基づきレコードごとに制御する     | 特定の従業員のみ同期、特定拠点にのみ配信             |
| 項目（カラム）単位 | 同期する項目を個別に指定する                 | ClassA は同期、ClassB は同期しない                   |
| 方向単位           | 同期方向とインスタンスの組み合わせで制御する | 子 A→親は ClassC を除外、子 B には ClassD を送らない |

### レコード単位の同期制御

サイト単位・カラム単位の制御に加え、レコード単位で同期対象・同期先を制御する仕組みを設ける。

#### ユースケース

| シナリオ                       | 制御例                                                     |
| ------------------------------ | ---------------------------------------------------------- |
| 本社専用データの配信除外       | 特定のレコードを同期対象外にする                           |
| 拠点別のデータ配信             | レコードの属性（地域コード等）に基づき特定サイトにのみ同期 |
| 機密データの同期制限           | 機密フラグ付きレコードを子インスタンスに同期しない         |
| 下書き・未承認データの配信停止 | Status が「承認済み」のレコードのみ同期する                |

#### 制御方式

レコードフィルタは SQL の `WHERE` 句として変換され、変更検出時に適用される。

| フィルタ種別      | 説明                                             | 例                                  |
| ----------------- | ------------------------------------------------ | ----------------------------------- |
| `include`         | 指定した条件に合致するレコードのみ同期する       | `ClassB = 'active'`                 |
| `exclude`         | 指定した条件に合致するレコードを同期対象外とする | `ClassC = 'confidential'`           |
| `targetOverrides` | 同期先ごとに異なるレコードフィルタを適用する     | 支社 A には地域コード = 'east' のみ |

```sql
-- レコードフィルタ適用例: 承認済みかつ機密でないレコードのみ同期
SELECT *
FROM "Results"
WHERE "SiteId" = @SiteId
    AND "UpdatedTime" > @LastSyncTime
    AND "Updator" <> @SyncUserId
    AND "ClassB" = 'approved'          -- include フィルタ
    AND "ClassC" <> 'confidential'     -- exclude フィルタ
ORDER BY "UpdatedTime" ASC

-- 同期先ごとのフィルタ例: 支社 A には地域コード 'east' のレコードのみ同期
SELECT *
FROM "Results"
WHERE "SiteId" = @SiteId
    AND "UpdatedTime" > @LastSyncTime
    AND "Updator" <> @SyncUserId
    AND "ClassD" = 'east'              -- targetOverride フィルタ
ORDER BY "UpdatedTime" ASC
```

> **注意**: レコードフィルタで同期対象外となったレコードが同期先に既に存在する場合、
> そのレコードを同期先から削除するかそのまま残すかは運用ポリシーで定める必要がある。

### 設定項目の説明

| 項目                              | 型     | 説明                                                 |
| --------------------------------- | ------ | ---------------------------------------------------- |
| `syncId`                          | string | 同期定義の一意識別子                                 |
| `topology`                        | string | `hub-spoke`（親子型）または `peer-to-peer`（対等型） |

| `source.dbms`                     | string | `SQLServer` / `PostgreSQL` / `MySQL`                 |
| `source.connectionString`         | string | DB 接続文字列                                        |
| `source.siteId`                   | number | 同期対象のサイト ID                                  |
| `targets`                         | array  | 同期先インスタンスの DB 接続情報の配列               |
| `syncKeys`                        | array  | レコード照合に使用するカラム名                       |
| `columns.default.include`         | array  | 既定で同期対象とするカラム一覧                       |
| `columns.overrides`               | array  | 特定インスタンス向けのカラム除外設定                 |
| `direction.sourceToTarget`        | bool   | 親→子方向の同期を有効にするか                        |
| `direction.targetToSource`        | object | 子→親方向の同期設定（有効フラグと除外カラム）        |
| `direction.overrides`             | array  | 特定インスタンスの方向制御を上書きする設定           |
| `changeDetection.method`          | string | 変更検出方式（後述）                                 |
| `changeDetection.intervalSeconds` | number | ポーリング間隔（秒）                                 |
| `conflictResolution`              | string | 競合時の解決方針                                     |
| `syncUser`                        | object | 同期処理で使用するプリザンターユーザ情報             |
| `recordFilter`                    | object | レコード単位の同期フィルタ定義                       |

| `recordFilter.exclude`            | object | 指定カラム値に合致するレコードを同期対象外とする     |
| `recordFilter.targetOverrides`    | array  | 同期先ごとのレコードフィルタ上書き設定               |
| `attachments`                     | object | 添付ファイル同期の設定                               |
| `attachments.enabled`             | bool   | 添付ファイルを同期対象に含めるか                     |
| `attachments.storageType`         | string | `Rds` のみサポート（DB 内保管の場合のみ同期可能）    |

## インスタンス間 ID マッピング

各プリザンターインスタンスは独立した DB を持つため、同一の実体であっても
各種 ID（SiteId・ResultId・UserId 等）がインスタンスごとに異なる。
同期アプリケーションではこの ID 差異を吸収するマッピング機構が必要となる。

### ID がインスタンス間で異なる項目

| ID 種別                | 説明                                | 同期への影響                                                 |
| ---------------------- | ----------------------------------- | ------------------------------------------------------------ |
| `SiteId`               | サイトの ID                         | 同期定義の `source.siteId` / `targets[].siteId` で紐付け済み |
| `ResultId`/`IssueId`   | レコードの主キー（自動採番）        | 同期キー（業務キー）で照合するため直接は使わない             |
| `WikiId`               | Wiki の主キー（自動採番）           | タイトルで照合または ID マッピングテーブルが必要             |
| `BinaryId`             | 添付ファイルの主キー（自動採番）    | 親レコードの同期キー + Guid で照合                           |
| `UserId`               | ユーザー ID                         | `Creator`/`Updator` のマッピングが必要                       |
| `DeptId`               | 部署 ID                             | Class カラムで部署を参照している場合に影響                   |
| `GroupId`              | グループ ID                         | Class カラムでグループを参照している場合に影響               |
| `Items.ReferenceId`    | Items の参照 ID（= レコード主キー） | レコード同期時に同期先の主キーで書き換える                   |
| `Binaries.ReferenceId` | 添付先レコードの ID                 | 同期先レコードの主キーに変換が必要                           |

### ID マッピングテーブル

同期アプリケーション専用のマッピングテーブルを用意し、インスタンス間の ID 対応を管理する。

```mermaid
erDiagram
    SyncIdMapping {
        string SyncId PK
        string EntityType PK
        string SyncKeyValue PK
        string SourceInstanceId
        bigint SourceId
        string TargetInstanceId
        bigint TargetId
        datetime CreatedTime
        datetime UpdatedTime
    }
    SyncUserMapping {
        string SourceInstanceId PK
        int SourceUserId PK
        string TargetInstanceId PK
        int TargetUserId
    }
```

| テーブル          | 用途                                                                      |
| ----------------- | ------------------------------------------------------------------------- |
| `SyncIdMapping`   | レコード ID（ResultId/IssueId/WikiId/BinaryId）のインスタンス間対応を保持 |
| `SyncUserMapping` | UserId のインスタンス間対応を保持（Creator/Updator の変換用）             |

### ID マッピングの処理フロー

```mermaid
flowchart TD
    A["同期元レコード取得"] --> B{"SyncIdMapping に
対応レコードが存在？"}
    B -->|Yes| C["マッピング済み TargetId を使用して UPDATE"]
    B -->|No| D["同期先で syncKey で検索"]
    D --> E{"syncKey で
レコード発見？"}
    E -->|Yes| F["既存レコードの TargetId を
マッピングに登録して UPDATE"]
    E -->|No| G["INSERT して新規採番された
TargetId をマッピングに登録"]

    C --> H["关連 ID の変換"]
    F --> H
    G --> H
    H --> I["Creator/Updator を
SyncUserMapping で変換"]
    I --> J["Items.ReferenceId を
同期先の主キーに設定"]
    J --> K["Binaries.ReferenceId を
同期先レコード ID に設定"]
```

### Creator / Updator の取り扱い

同期書き込み時、`Creator` と `Updator` は以下のいずれかの方針で処理する。

| 方針                        | 説明                                                   | 適用場面                   |
| --------------------------- | ------------------------------------------------------ | -------------------------- |
| 同期ユーザーで固定          | 常に `syncUser.userId` を設定する                      | 簡易運用・履歴不要の場合   |
| SyncUserMapping で変換      | 同期元の UserId を同期先の対応 UserId に変換する       | 操作者情報を保持したい場合 |
| Creator 保持 + Updator 固定 | 初回同期時はマッピング変換、更新時は同期ユーザーを設定 | ループ防止と履歴保持の両立 |

> **推奨**: 同期ループ防止のため `Updator` は同期専用ユーザー ID で固定し、
> `Creator` は SyncUserMapping で変換する方針を推奨する。
> ユーザーマッピングが未定義の場合は同期ユーザー ID にフォールバックする。

### Class カラム内の ID 参照

Class カラムに他テーブルのレコード ID（リンク）やユーザー ID・部署 ID・グループ ID が
格納されている場合、それらの値もインスタンス間で異なる。

| ケース                       | 例                   | 対応策                                               |
| ---------------------------- | -------------------- | ---------------------------------------------------- |
| 選択肢リンク（他サイト参照） | `ClassA = "[12345]"` | SyncIdMapping でリンク先 ID を同期先の ID に変換     |
| ユーザー選択（`[[Users]]`）  | `ClassB = "42"`      | SyncUserMapping で変換                               |
| 部署選択（`[[Depts]]`）      | `ClassC = "5"`       | 部署マッピングテーブルを追加するか、部署コードで照合 |
| グループ選択（`[[Groups]]`） | `ClassD = "3"`       | グループマッピングを追加するか、グループ名で照合     |

> **重要**: 同期定義の `columns` 設定で、ID 参照を含むカラムには
> 変換ルール（どのマッピングテーブルを使うか）を明示的に指定する必要がある。
> ID 参照の変換が不可能な場合（マッピング未登録）は警告ログを出力し、
> 元の値をそのまま書き込むかエラーとするかを設定で選択できるようにする。

## 変更検出方式

リアルタイム同期を実現するため、RDBMS ごとの変更検出方式を検討する。

### RDBMS 別の変更検出機能

| 方式                | SQL Server          | PostgreSQL             | MySQL               |
| ------------------- | ------------------- | ---------------------- | ------------------- |
| ポーリング          | ✅ UpdatedTime 比較 | ✅ UpdatedTime 比較    | ✅ UpdatedTime 比較 |
| Change Data Capture | ✅ CDC              | ✅ Logical Replication | ✅ Binlog           |
| トリガー            | ✅ DML Trigger      | ✅ NOTIFY/LISTEN       | ✅ DML Trigger      |
| 変更追跡            | ✅ Change Tracking  | ❌                     | ❌                  |

### 方式 1: ポーリング方式（全 RDBMS 共通・推奨）

`UpdatedTime` カラムを条件に定期的に変更レコードを取得する方式。
全 RDBMS で同一ロジックで実装できるため、推奨方式とする。

```mermaid
sequenceDiagram
    participant Sync as 同期アプリ
    participant SrcDB as 同期元 DB
    participant DstDB as 同期先 DB

    loop ポーリング間隔ごと（例: 5秒）
        Sync->>SrcDB: 前回同期以降の変更を取得<br/>WHERE UpdatedTime > @lastSync
        SrcDB-->>Sync: 変更レコード一覧
        alt 変更あり
            Sync->>Sync: 同期ルール適用<br/>（カラムフィルタ・方向制御）
            Sync->>DstDB: BEGIN TRANSACTION
            Sync->>DstDB: 同期先にレコード存在確認<br/>WHERE syncKey = @key
            alt レコード存在
                Sync->>DstDB: UPDATE（対象カラムのみ）
            else レコード不在
                Sync->>DstDB: INSERT（Results + Items）
            end
            Sync->>DstDB: COMMIT
        end
        Sync->>Sync: lastSyncTime を更新
    end
```

**変更取得 SQL（PostgreSQL の例）**:

```sql
-- Results テーブルから変更レコードを取得
SELECT
    "ResultId", "SiteId", "Title", "Body",
    "ClassA", "ClassB", "ClassC",
    "NumA", "NumB",
    "DateA",
    "Ver", "Creator", "Updator",
    "CreatedTime", "UpdatedTime"
FROM "Results"
WHERE "SiteId" = @SiteId
    AND "UpdatedTime" > @LastSyncTime
    AND "Updator" <> @SyncUserId
ORDER BY "UpdatedTime" ASC
```

**同期先への書き込み SQL（PostgreSQL の例）**:

```sql
-- 同期先にレコードが存在するか確認
SELECT "ResultId", "Ver", "UpdatedTime"
FROM "Results"
WHERE "SiteId" = @TargetSiteId AND "ClassA" = @SyncKeyValue

-- 存在する場合: UPDATE
UPDATE "Results" SET
    "Title" = @Title,
    "ClassB" = @ClassB,
    "ClassC" = @ClassC,
    "NumA" = @NumA,
    "Body" = @Body,
    "Ver" = "Ver" + 1,
    "Updator" = @SyncUserId,
    "UpdatedTime" = CURRENT_TIMESTAMP
WHERE "SiteId" = @TargetSiteId AND "ClassA" = @SyncKeyValue

-- Items テーブルも更新
UPDATE "Items" SET
    "Title" = @Title,
    "Updator" = @SyncUserId,
    "UpdatedTime" = CURRENT_TIMESTAMP
WHERE "SiteId" = @TargetSiteId
    AND "ReferenceId" = @ResultId
    AND "ReferenceType" = 'Results'

-- 存在しない場合: INSERT
INSERT INTO "Results" (
    "SiteId", "Title", "Body",
    "ClassA", "ClassB", "ClassC", "NumA",
    "Ver", "Creator", "Updator",
    "CreatedTime", "UpdatedTime"
) VALUES (
    @TargetSiteId, @Title, @Body,
    @SyncKeyValue, @ClassB, @ClassC, @NumA,
    1, @SyncUserId, @SyncUserId,
    CURRENT_TIMESTAMP, CURRENT_TIMESTAMP
)

-- Items テーブルにも INSERT
INSERT INTO "Items" (
    "ReferenceId", "ReferenceType", "SiteId",
    "Title", "Updator", "UpdatedTime"
) VALUES (
    @NewResultId, 'Results', @TargetSiteId,
    @Title, @SyncUserId, CURRENT_TIMESTAMP
)
```

| メリット                         | デメリット                       |
| -------------------------------- | -------------------------------- |
| 全 RDBMS で同一ロジック          | ポーリング間隔分の遅延が発生する |
| RDBMS 固有の設定が不要           | 短間隔ポーリングで DB 負荷が増加 |
| 実装が単純で保守しやすい         | 削除の検出に追加処理が必要       |
| プリザンター DB への影響が最小限 |                                  |

### 方式 2: CDC / Binlog / Logical Replication 方式

RDBMS 固有の変更データキャプチャ機能を使用して
リアルタイムに変更を検出する方式。

#### SQL Server: Change Data Capture（CDC）

```sql
-- CDC を有効化（DBA 作業）
EXEC sys.sp_cdc_enable_db
EXEC sys.sp_cdc_enable_table
    @source_schema = N'dbo',
    @source_name = N'Results',
    @role_name = NULL,
    @supports_net_changes = 1

-- 変更データの取得
DECLARE @from_lsn binary(10) =
    sys.fn_cdc_get_min_lsn('dbo_Results')
DECLARE @to_lsn binary(10) =
    sys.fn_cdc_get_max_lsn()
SELECT * FROM cdc.fn_cdc_get_net_changes_dbo_Results(
    @from_lsn, @to_lsn, 'all'
)
```

#### PostgreSQL: Logical Replication

```sql
-- 論理レプリケーションスロットの作成（DBA 作業）
SELECT pg_create_logical_replication_slot(
    'pleasanter_sync', 'pgoutput'
);

-- パブリケーションの作成
CREATE PUBLICATION pleasanter_sync_pub
FOR TABLE "Results", "Items"
WHERE ("SiteId" IN (対象SiteId一覧));
```

アプリケーション側では `pgoutput` プロトコルを使用して
変更ストリームを受信する。

#### MySQL: Binary Log（Binlog）

```sql
-- Binlog の有効化（my.cnf）
-- server-id=1
-- log-bin=mysql-bin
-- binlog-format=ROW
-- binlog-row-image=FULL

-- Binlog イベントの取得
SHOW BINARY LOG STATUS;
-- アプリケーション側で binlog クライアント
-- ライブラリを使用してイベントを受信する
```

| メリット                       | デメリット                                |
| ------------------------------ | ----------------------------------------- |
| ほぼリアルタイムの変更検出     | RDBMS ごとに実装が異なる                  |
| ポーリング不要で DB 負荷が低い | DBA による事前設定が必要                  |
| 削除も含めて全変更を検出可能   | CDC/Binlog の管理（容量・保持期間）が必要 |
|                                | 既存環境への設定変更が必要                |

### 方式 3: ハイブリッド方式（推奨）

ポーリングをベースとし、利用可能な環境では
CDC/Binlog/Logical Replication を併用する方式。

```mermaid
graph TB
    subgraph "変更検出層"
        Polling["ポーリング<br/>（全 RDBMS 共通）<br/>5秒間隔"]
        CDC["CDC / Binlog /<br/>Logical Replication<br/>（利用可能な場合）"]
    end

    subgraph "同期処理層"
        Queue["変更キュー"]
        Filter["同期ルール適用<br/>（カラム・方向フィルタ）"]
        Writer["DB 書き込み<br/>（トランザクション）"]
    end

    Polling -->|変更検出| Queue
    CDC -->|変更検出| Queue
    Queue --> Filter
    Filter --> Writer
    Writer --> TargetDB["同期先 DB"]
```

| 環境                | 動作モード                               |
| ------------------- | ---------------------------------------- |
| CDC/Binlog 設定済み | CDC/Binlog で即時検出 + バッチ整合性検証 |
| CDC/Binlog 未設定   | ポーリングのみで動作（フォールバック）   |

## 削除の同期

プリザンターはレコード削除時に `_deleted` テーブルにレコードをコピーする。
この仕組みを利用して削除を検出する。

### 削除検出の SQL

```sql
-- _deleted テーブルから削除されたレコードを検出
SELECT
    "ResultId", "SiteId", "ClassA",
    "UpdatedTime" AS "DeletedTime"
FROM "Results_deleted"
WHERE "SiteId" = @SiteId
    AND "UpdatedTime" > @LastSyncTime
ORDER BY "UpdatedTime" ASC
```

### 削除同期の処理フロー

```mermaid
sequenceDiagram
    participant Sync as 同期アプリ
    participant SrcDB as 同期元 DB
    participant DstDB as 同期先 DB

    Sync->>SrcDB: Results_deleted から削除レコード取得
    SrcDB-->>Sync: 削除レコード一覧
    loop 各削除レコード
        Sync->>DstDB: BEGIN TRANSACTION
        Sync->>DstDB: 同期先の対応レコードを検索<br/>WHERE syncKey = @key
        alt レコード存在
            Sync->>DstDB: Results_deleted に INSERT<br/>（プリザンターの削除処理を模倣）
            Sync->>DstDB: Results から DELETE
            Sync->>DstDB: Items から DELETE
        end
        Sync->>DstDB: COMMIT
    end
```

> **注意**: 同期先でのレコード削除はプリザンターの削除フロー
> （base → \_deleted へコピー → base から削除）を忠実に再現する必要がある。
> 直接 DELETE のみを実行すると `_deleted` テーブルとの整合性が崩れる。

## 競合解決の方針

対等型トポロジでは、同一レコードが複数インスタンスで同時に編集される可能性がある。

### 競合解決戦略

| 戦略              | 説明                                     | 適用場面                     |
| ----------------- | ---------------------------------------- | ---------------------------- |
| Source Wins       | 同期元（親）のデータを常に優先する       | 親子型で正本が明確な場合     |
| Last Write Wins   | `UpdatedTime` が新しい方を採用する       | 対等型で更新頻度が低い場合   |
| Manual Resolution | 競合を検出し管理者に通知して手動解決する | データの正確性が最重要の場合 |
| Field-Level Merge | カラム単位で最終更新を比較しマージする   | 異なるカラムの同時編集時     |

### 競合検出の実装

```mermaid
flowchart TD
    A["同期元の変更レコード取得"] --> B{"同期先に<br/>同じ syncKey の<br/>レコードが存在？"}
    B -->|No| C["INSERT（新規作成）"]
    B -->|Yes| D{"同期元 UpdatedTime<br/>><br/>同期先 UpdatedTime？"}
    D -->|Yes| E["UPDATE（上書き）"]
    D -->|No| F{"同期先 UpdatedTime<br/>><br/>前回同期時刻？"}
    F -->|No| G["スキップ（変更なし）"]
    F -->|Yes| H["競合検出<br/>（解決戦略を適用）"]
```

## 同期ループの防止

双方向同期では、同期による更新が再び変更として検出され
無限ループとなるリスクがある。

```mermaid
sequenceDiagram
    participant A as インスタンス A
    participant Sync as 同期アプリ
    participant B as インスタンス B

    Note over A: ユーザが更新
    Sync->>A: 変更検出
    Sync->>B: 同期書き込み
    Note over B: UpdatedTime 更新
    Sync->>B: 変更検出（ループ！）
    Sync->>A: 同期書き込み（ループ！）
```

### 防止策

| 方式             | 説明                                                       |
| ---------------- | ---------------------------------------------------------- |
| 同期ユーザ判定   | `Updator` が同期専用ユーザ ID の場合は同期対象外とする     |
| 書き込み履歴管理 | 同期アプリ側で書き込み済みレコードを記録し再検出を抑止する |
| UpdatedTime 固定 | 同期書き込み時に元の `UpdatedTime` をそのまま使用する      |

**推奨**: 同期ユーザ判定方式。同期アプリ専用のプリザンターユーザ（例: `SyncService`）を
作成し、`Updator` がそのユーザ ID のレコードは変更検出から除外する。

```sql
-- 変更検出時に同期ユーザの更新を除外
SELECT * FROM "Results"
WHERE "SiteId" = @SiteId
    AND "UpdatedTime" > @LastSyncTime
    AND "Updator" <> @SyncUserId
ORDER BY "UpdatedTime" ASC
```

| 要素              | 候補                                                   |
| ----------------- | ------------------------------------------------------ |
| 言語              | C#（.NET 8+）                                          |
| SQL Server 接続   | Microsoft.Data.SqlClient                               |
| PostgreSQL 接続   | Npgsql                                                 |
| MySQL 接続        | MySqlConnector                                         |
| 設定管理          | Microsoft.Extensions.Configuration（JSON ファイル）    |
| ログ              | NLog または Serilog                                    |
| ホスティング      | .NET Generic Host（Windows Service / systemd 対応）    |
| CDC（SQL Server） | SqlClient での CDC クエリ                              |
| CDC（PostgreSQL） | Npgsql.Replication（Logical Replication クライアント） |
| CDC（MySQL）      | MySqlCdc（Binlog クライアントライブラリ）              |

> C# を選定する理由: プリザンター本体が C#（.NET）で実装されており、
> DB アクセスライブラリやデータ型の取り扱いに親和性が高いため。

### 処理フロー

```mermaid
flowchart TD
    Start["アプリケーション起動"] --> LoadConfig["設定ファイル読み込み"]
    LoadConfig --> ConnectDB["全インスタンスの DB に接続"]
    ConnectDB --> DetectType["Sites.ReferenceType を確認<br/>（Results / Issues / Wikis）"]
    DetectType --> MainLoop["メインループ開始"]

    MainLoop --> PollChanges["変更検出<br/>（UpdatedTime > lastSync）"]
    PollChanges --> HasChanges{"変更あり？"}
    HasChanges -->|No| WaitInterval["インターバル待機"]
    HasChanges -->|Yes| ApplyRules["同期ルール適用<br/>（カラム・レコード・方向フィルタ）"]
    ApplyRules --> CheckConflict{"競合あり？"}
    CheckConflict -->|No| WriteTarget["同期先 DB に書き込み<br/>（トランザクション）"]
    CheckConflict -->|Yes| ResolveConflict["競合解決戦略を適用"]
    ResolveConflict --> WriteTarget
    WriteTarget --> UpdateItems["Items テーブルを更新"]
    UpdateItems --> SyncBinaries["Binaries 同期<br/>（添付ファイルがある場合）"]
    SyncBinaries --> LogResult["同期ログ記録"]
    LogResult --> PollDeleted["削除検出<br/>（_deleted テーブル）"]
    PollDeleted --> SyncDeleted["削除の同期"]
    SyncDeleted --> WaitInterval
    WaitInterval --> PollChanges
```

## RDBMS 別の SQL 差異

同期アプリケーションが生成する SQL は RDBMS によって構文が異なる。

### 識別子のクォート

| RDBMS      | クォート文字 | 例                       |
| ---------- | ------------ | ------------------------ |
| SQL Server | `[]`         | `[Results].[ClassA]`     |
| PostgreSQL | `""`         | `"Results"."ClassA"`     |
| MySQL      | `` ` ` ``    | `` `Results`.`ClassA` `` |

### INSERT 時の自動採番 ID 取得

| RDBMS      | 方式                       |
| ---------- | -------------------------- |
| SQL Server | `OUTPUT INSERTED.ResultId` |
| PostgreSQL | `RETURNING "ResultId"`     |
| MySQL      | `LAST_INSERT_ID()`         |

### 日時関数

| RDBMS      | 現在日時の取得                |
| ---------- | ----------------------------- |
| SQL Server | `GETDATE()` / `SYSDATETIME()` |
| PostgreSQL | `CURRENT_TIMESTAMP`           |
| MySQL      | `NOW()`                       |

### UPSERT 構文（参考）

各 RDBMS にはネイティブの UPSERT 構文があるが、同期処理では
Items テーブルとの整合性制御が必要なため、
SELECT → INSERT/UPDATE の明示的な分岐を推奨する。

| RDBMS      | UPSERT 構文                            |
| ---------- | -------------------------------------- |
| SQL Server | `MERGE ... WHEN MATCHED / NOT MATCHED` |
| PostgreSQL | `INSERT ... ON CONFLICT DO UPDATE`     |
| MySQL      | `INSERT ... ON DUPLICATE KEY UPDATE`   |

## セキュリティに関する考慮事項

| 項目              | 対策                                                          |
| ----------------- | ------------------------------------------------------------- |
| DB 接続情報の管理 | 接続文字列は暗号化して保存する（DPAPI / Key Vault 等）        |
| DB アクセス権限   | 同期対象テーブルのみに SELECT/INSERT/UPDATE/DELETE を付与する |
| 通信経路の暗号化  | DB 接続に TLS を使用する                                      |
| 同期ユーザの分離  | 同期専用のプリザンターユーザ・DB ユーザを作成する             |
| ログ出力          | 接続文字列やレコード内容をログに出力しない                    |
| 監査              | 同期操作の履歴を保持し異常を検出可能にする                    |

## 運用上の注意事項

### サイト定義（SiteSettings）の同期

本設計はレコードデータの同期を対象とする。サイト定義（カラム設定・ビュー設定等）の同期は
対象外であり、各インスタンスのサイト定義は事前に手動で整合させておく必要がある。

### Ver カラムの取り扱い

プリザンターの `Ver` カラムは楽観的排他制御に使用される。
同期書き込み時には `Ver` をインクリメントして更新する。

```sql
UPDATE "Results" SET
    "Title" = @Title,
    "ClassA" = @ClassA,
    "Ver" = "Ver" + 1,
    "Updator" = @SyncUserId,
    "UpdatedTime" = CURRENT_TIMESTAMP
WHERE "SiteId" = @SiteId AND "ClassA" = @SyncKeyValue
```

### History テーブルの取り扱い

プリザンターはレコード更新時に `_history` テーブルにバージョン履歴を保存する。
同期書き込み時にもこのパターンを再現する場合は、以下の SQL を追加する。

```sql
-- 更新前のレコードを _history にコピー
INSERT INTO "Results_history" (
    "ResultId", "Ver", "SiteId", "Title", "Body",
    "ClassA", "ClassB",
    "Creator", "Updator", "CreatedTime", "UpdatedTime"
)
SELECT
    "ResultId", "Ver", "SiteId", "Title", "Body",
    "ClassA", "ClassB",
    "Creator", "Updator", "CreatedTime", "UpdatedTime"
FROM "Results"
WHERE "SiteId" = @SiteId AND "ClassA" = @SyncKeyValue
```

> **注意**: History テーブルへの書き込みは必須ではないが、
> プリザンター上でのバージョン履歴表示との整合性を保つためには推奨される。

### 同期キーの設計

| 注意点         | 説明                                                               |
| -------------- | ------------------------------------------------------------------ |
| キーの一意性   | 同期キーは全インスタンスで一意になるよう設計する                   |
| 自動採番の回避 | ResultId / IssueId / WikiId はインスタンスごとに異なるため使用不可 |
| 業務キーの使用 | 従業員番号・商品コードなど業務上の一意キーを使用する               |

## Wikis テーブルの同期

Wiki ページ（`ReferenceType = 'Wikis'`）も同期対象とする。
Wikis テーブルは Results / Issues と似た構造だが、以下の点が異なる。

### Wikis と Results / Issues の構造差異

| 項目               | Results / Issues             | Wikis                                     |
| ------------------ | ---------------------------- | ----------------------------------------- |
| 主キー             | `ResultId` / `IssueId`       | `WikiId`                                  |
| Class/Num/Date     | `ClassA`〜`ClassZ` 等あり    | なし（Title と Body のみ）                |
| Locked             | なし                         | `Locked`（ロック状態）                    |
| 同期キー候補       | Class カラム（業務キー）     | `Title`（サイト内で一意とする運用が前提） |
| テーブルバリアント | `_deleted` / `_history` あり | `Wikis_deleted` / `Wikis_history` あり    |
| レコード数         | 1 サイトに複数レコード       | 通常 1 サイトに 1 レコード（Wiki ページ） |

### Wiki 同期時の考慮事項

| 考慮事項                  | 対応                                                                 |
| ------------------------- | -------------------------------------------------------------------- |
| 同期キー                  | Wiki には Class カラムがないため、`Title` または ID マッピングで照合 |
| SiteId の対応             | Wiki サイト自体の SiteId がインスタンス間で異なる                    |
| Body 内の ID 参照         | Markdown 内のリンク（`/items/12345`）に含まれる ID の変換が必要      |
| 添付ファイル              | Wiki の添付ファイルも Binaries テーブルで管理される                  |
| 1 サイト 1 レコードの原則 | Wiki は通常 1 サイト = 1 ページなので、サイト単位で同期定義を作る    |

### Wiki 同期の SQL 例（PostgreSQL）

```sql
-- Wiki の変更検出
SELECT
    "WikiId", "SiteId", "Title", "Body",
    "Locked", "Ver",
    "Creator", "Updator",
    "CreatedTime", "UpdatedTime"
FROM "Wikis"
WHERE "SiteId" = @SiteId
    AND "UpdatedTime" > @LastSyncTime
    AND "Updator" <> @SyncUserId
ORDER BY "UpdatedTime" ASC

-- Wiki の同期先への書き込み（UPDATE）
UPDATE "Wikis" SET
    "Title" = @Title,
    "Body" = @Body,
    "Locked" = @Locked,
    "Ver" = "Ver" + 1,
    "Updator" = @SyncUserId,
    "UpdatedTime" = CURRENT_TIMESTAMP
WHERE "SiteId" = @TargetSiteId AND "WikiId" = @TargetWikiId

-- Items テーブルも更新
UPDATE "Items" SET
    "Title" = @Title,
    "Updator" = @SyncUserId,
    "UpdatedTime" = CURRENT_TIMESTAMP
WHERE "ReferenceId" = @TargetWikiId
    AND "ReferenceType" = 'Wikis'
```

## 添付ファイル（Binaries）の同期

レコードに紐付く添付ファイルも同期対象とする。
**RDS（DB 内）保管の場合のみ**同期可能であり、ファイルシステム保管の場合は対象外とする。

### Binaries テーブルの同期対象カラム

| カラム        | 同期説明                                                |
| ------------- | ------------------------------------------------------- |
| `BinaryId`    | インスタンスごとに自動採番→ID マッピングで管理          |
| `TenantId`    | 同期先の TenantId に変換                                |
| `ReferenceId` | 同期先レコードの主キー（ResultId/IssueId/WikiId）に変換 |
| `Guid`        | そのまま保持（ファイル識別子として同期キーにも使用）    |
| `BinaryType`  | そのままコピー（`Attachments` / `Images`）              |
| `Title`       | そのままコピー                                          |
| `Bin`         | バイナリデータをそのままコピー（RDS 保管時のみ）        |
| `Thumbnail`   | そのままコピー                                          |
| `Icon`        | そのままコピー                                          |
| `FileName`    | そのままコピー                                          |
| `Extension`   | そのままコピー                                          |
| `Size`        | そのままコピー                                          |
| `ContentType` | そのままコピー                                          |

### 添付ファイル同期の前提条件

| 条件               | 説明                                                                   |
| ------------------ | ---------------------------------------------------------------------- |
| 保管方式 = RDS     | `BinaryStorage.Provider` が `"Rds"` の環境のみ同期可能                 |
| 親レコード同期済み | 添付先の Results/Issues/Wikis レコードが先に同期されていること         |
| ID マッピング済み  | 親レコードの ID マッピングが確立していること（ReferenceId 変換に必要） |

### 添付ファイル同期の処理フロー

```mermaid
sequenceDiagram
    participant Sync as 同期アプリ
    participant SrcDB as 同期元 DB
    participant DstDB as 同期先 DB

    Note over Sync: 親レコードの同期完了後に実行
    Sync->>SrcDB: 親レコードの Binaries を取得<br/>WHERE ReferenceId = @SourceRecordId<br/>AND BinaryType IN ('Attachments','Images')
    SrcDB-->>Sync: 添付ファイル一覧（Bin 含む）

    loop 各添付ファイル
        Sync->>DstDB: Guid で同期先の既存 Binary を検索
        alt 既存あり
            Sync->>DstDB: UPDATE Binaries<br/>（Bin・FileName・Size 等を更新）
        else 既存なし
            Sync->>DstDB: INSERT Binaries<br/>（ReferenceId = 同期先レコード ID）
        end
    end

    Note over Sync: 同期元で削除された添付を検出
    Sync->>DstDB: 同期先にあるが同期元にない<br/>Guid の Binaries を DELETE
```

### 添付ファイル同期の SQL 例（PostgreSQL）

```sql
-- 同期元の添付ファイルを取得
SELECT
    "BinaryId", "TenantId", "ReferenceId",
    "Guid", "BinaryType", "Title",
    "Bin", "Thumbnail", "Icon",
    "FileName", "Extension", "Size", "ContentType",
    "CreatedTime", "UpdatedTime"
FROM "Binaries"
WHERE "ReferenceId" = @SourceRecordId
    AND "BinaryType" IN ('Attachments', 'Images')

-- 同期先に Guid で既存確認
SELECT "BinaryId"
FROM "Binaries"
WHERE "ReferenceId" = @TargetRecordId
    AND "Guid" = @Guid

-- 既存の場合: UPDATE
UPDATE "Binaries" SET
    "Bin" = @Bin,
    "Thumbnail" = @Thumbnail,
    "Icon" = @Icon,
    "FileName" = @FileName,
    "Extension" = @Extension,
    "Size" = @Size,
    "ContentType" = @ContentType,
    "Updator" = @SyncUserId,
    "UpdatedTime" = CURRENT_TIMESTAMP
WHERE "ReferenceId" = @TargetRecordId AND "Guid" = @Guid

-- 新規の場合: INSERT
INSERT INTO "Binaries" (
    "TenantId", "ReferenceId", "Guid", "BinaryType",
    "Title", "Bin", "Thumbnail", "Icon",
    "FileName", "Extension", "Size", "ContentType",
    "Creator", "Updator", "CreatedTime", "UpdatedTime"
) VALUES (
    @TargetTenantId, @TargetRecordId, @Guid, @BinaryType,
    @Title, @Bin, @Thumbnail, @Icon,
    @FileName, @Extension, @Size, @ContentType,
    @SyncUserId, @SyncUserId, CURRENT_TIMESTAMP, CURRENT_TIMESTAMP
)

-- 同期元で削除された添付ファイルを同期先からも削除
DELETE FROM "Binaries"
WHERE "ReferenceId" = @TargetRecordId
    AND "Guid" NOT IN (@SyncedGuidList)
    AND "BinaryType" IN ('Attachments', 'Images')
```

### 添付ファイル同期の注意事項

| 項目                    | 説明                                                                    |
| ----------------------- | ----------------------------------------------------------------------- |
| データサイズ            | 大きな添付ファイルの同期は DB 負荷・ネットワーク帯域に影響する          |
| `Bin` のバッチサイズ    | バイナリデータの転送はチャンク分割を検討する                            |
| ストレージ方式の差異    | 同期元・Sync 先の両方が `Rds` 保管であることを事前に確認する            |
| 削除検出                | Binaries に `_deleted` テーブルがないため、Guid の差分比較で削除を検出  |
| Body 内の画像参照       | Markdown/HTML の `![](...)` 内の Guid 参照は変換不要（Guid 保持のため） |
| Attachment JSON の Guid | Class/Description カラム内の添付 JSON に含まれる Guid も保持される      |

## 結論

| 項目             | 結論                                                                                      |
| ---------------- | ----------------------------------------------------------------------------------------- |
| 実装形態         | プリザンターとは独立した外部アプリケーションとして実装する                                |
| データアクセス   | API を使用せず RDBMS を直接操作する                                                       |
| 対応 RDBMS       | SQL Server・MySQL・PostgreSQL の 3 種に対応する                                           |
| 推奨言語         | C#（.NET 8+）。プリザンターとの技術的親和性が高い                                         |
| 変更検出         | ポーリング方式（`UpdatedTime` 比較）を基本とし、CDC/Binlog が利用可能な環境では併用する   |
| 推奨トポロジ     | 親子型（Hub-Spoke）を推奨。競合リスクが低く実装が容易                                     |
| 同期制御         | サイト単位・レコード単位・カラム単位・方向単位の 4 レベルで制御可能                       |
| 同期キー         | 業務上の一意キーを Class カラムに格納して照合に使用する                                   |
| ID マッピング    | インスタンス間で異なる各種 ID（SiteId・レコード ID・UserId 等）をマッピングテーブルで管理 |
| 競合解決         | 親子型では Source Wins、対等型では Last Write Wins を基本とする                           |
| 削除検出         | `_deleted` テーブルを直接参照して削除されたレコードを検出する                             |
| 同期ループ防止   | 同期専用ユーザの `Updator` 判定で防止する                                                 |
| Items テーブル   | Results/Issues/Wikis と Items の両テーブルをトランザクション内で整合的に更新する          |
| Wikis 同期       | Wiki ページも同期対象。Title または ID マッピングで照合する                               |
| 添付ファイル同期 | Binaries テーブル（RDS 保管時のみ）を Guid で照合して同期する                             |
| Ver/History      | `Ver` をインクリメントし、必要に応じて `_history` テーブルにも書き込む                    |
| リアルタイム性   | ポーリング間隔 5 秒程度でほぼリアルタイムの同期を実現可能                                 |

## 注意事項

- DB 直接操作はプリザンターのアプリケーションレベルのバリデーション
  （権限チェック・入力検証等）をバイパスするため、
  同期データの整合性は同期アプリケーション側で担保する必要がある
- プリザンターのバージョンアップでテーブル構造が変更された場合、
  同期アプリケーションの対応が必要となる
- 同期対象テーブルにインデックスが適切に設定されていることを確認する
  （特に `UpdatedTime` と同期キーカラムへのインデックス）
- 大量レコードの初回同期はバッチサイズを分割して実行する
