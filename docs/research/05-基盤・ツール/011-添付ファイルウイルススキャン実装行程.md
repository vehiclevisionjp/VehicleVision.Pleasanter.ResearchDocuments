### スキャンモード（ScanMode）

`VirusScan.json` の `ScanMode` フィールドで 3 パターンのスキャン対象を制御する。

| ScanMode     | 値  | 対象コントローラー                                                   | 用途                                     |
| ------------ | :-: | -------------------------------------------------------------------- | ---------------------------------------- |
| `"All"`      |  0  | BinariesController + FormBinariesController + Api/BinariesController | 全アップロード経路をスキャン（最も安全） |
| `"FormOnly"` |  1  | FormBinariesController のみ                                          | 外部公開フォーム経由のみスキャン（軽量） |
| `"Disabled"` |  2  | なし                                                                 | スキャン無効（従来動作）                 |

#### 各モードの対象マッピング

| アップロード経路                   | All | FormOnly | Disabled |
| ---------------------------------- | :-: | :------: | :------: |
| BinariesController.Upload          | 可  |   不可   |   不可   |
| BinariesController.UploadImage     | 可  |   不可   |   不可   |
| FormBinariesController.Upload      | 可  |    可    |   不可   |
| FormBinariesController.UploadImage | 可  |    可    |   不可   |
| Api/BinariesController.Upload      | 可  |   不可   |   不可   |

#### 判定ロジック

`BinaryUtilities` のスキャンヘルパーに `isFormUpload` 引数を追加し、
呼び出し元のコントローラー種別に応じてスキャン実行を判定する。

```csharp
public enum VirusScanMode
{
    All = 0,
    FormOnly = 1,
    Disabled = 2
}

private static bool ShouldScan(bool isFormUpload)
{
    var mode = Parameters.VirusScan?.ScanMode ?? VirusScanMode.Disabled;
    return mode switch
    {
        VirusScanMode.All => true,
        VirusScanMode.FormOnly => isFormUpload,
        _ => false
    };
}
```

#### 使い分けの指針

| シナリオ                                           | 推奨モード |
| -------------------------------------------------- | ---------- |
| 社内利用のみ（イントラネット）                     | `Disabled` |
| 外部公開フォームあり・内部ユーザーは信頼           | `FormOnly` |
| 外部公開フォーム + API 公開 + 不特定多数がアクセス | `All`      |
| ClamAV サーバーのリソースが限られている            | `FormOnly` |

namespace Implem.ParameterAccessor.Parts
{
    public class VirusScan
    {
        /// <summary>スキャンモード (All / FormOnly / Disabled)</summary>
        public VirusScanMode ScanMode;

        /// <summary>スキャンプロバイダー ("ClamAV")</summary>
        public string Provider;

#### Step 2-3: ClamAV 実装

**ファイル**: `Implem.Pleasanter/Libraries/Security/ClamAvVirusScanService.cs`（新規）

```csharp
using nClam;
using Implem.ParameterAccessor;

namespace Implem.Pleasanter.Libraries.Security
{
    public class ClamAvVirusScanService : IVirusScanService
    {
        private readonly ClamClient _client;
        private readonly int _maxFileSizeBytes;

        public ClamAvVirusScanService()
        {
            var host = Parameters.VirusScan?.ClamAvHost ?? "localhost";
            var port = Parameters.VirusScan?.ClamAvPort ?? 3310;
            var timeoutMs = Parameters.VirusScan?.TimeoutMs ?? 30000;
            _maxFileSizeBytes = (Parameters.VirusScan?.MaxFileSizeMB ?? 100) * 1024 * 1024;
            _client = new ClamClient(host, port)
            {
                MaxStreamSize = _maxFileSizeBytes
            };
        }

        public async Task<VirusScanResult> ScanAsync(byte[] data, string fileName)
        {
            if (data == null || data.Length == 0)
                return VirusScanResult.Clean();

            if (data.Length > _maxFileSizeBytes)
                return VirusScanResult.Clean(); // サイズ超過はスキップ

            try
            {
                var result = await _client.SendAndScanFileAsync(data);
                return result.Result switch
                {
                    ClamScanResults.Clean => VirusScanResult.Clean(),
                    ClamScanResults.VirusDetected => VirusScanResult.Infected(
                        result.InfectedFiles?.FirstOrDefault()?.VirusName
                            ?? "Unknown"),
                    _ => VirusScanResult.Error(
                        $"ClamAV scan returned: {result.Result}")
                };
            }
            catch (Exception ex)
            {
                return VirusScanResult.Error(
                    $"ClamAV connection error: {ex.Message}");
            }
        }
    }
}
```

### Phase 3: アップロードフローへの統合

#### Step 3-1: サービスロケーターによるアクセス

`BinaryUtilities` は static クラスであり DI を直接受けられない。
Pleasanter の既存パターンに合わせて `AspNetCoreHttpContext.Current.RequestServices`
経由でサービスを取得するヘルパーメソッドを追加する。

**ファイル**: `Implem.Pleasanter/Models/Binaries/BinaryUtilities.cs` — ヘルパーメソッド追加

```csharp
using AspNetCoreCurrentRequestContext;
using Implem.Pleasanter.Libraries.Security;
using Microsoft.Extensions.DependencyInjection;

private static IVirusScanService GetVirusScanService()
{
    return AspNetCoreHttpContext.Current?
        .RequestServices?
        .GetService<IVirusScanService>()
            ?? new NullVirusScanService();
}

/// <summary>
/// アップロードされたファイルのウイルススキャンを実行する。
/// ScanMode と isFormUpload の組み合わせでスキャン要否を判定する。
/// </summary>
private static async Task<VirusScanResult> ScanFileAsync(
    byte[] data,
    string fileName,
    bool isFormUpload = false)
{
    if (!ShouldScan(isFormUpload))
        return VirusScanResult.Clean();

    var scanner = GetVirusScanService();
    return await scanner.ScanAsync(data, fileName);
}
```

**挿入コード概要**:

`UploadFile` は `BinariesController` と `FormBinariesController` の両方から
呼ばれるため、呼び出し元の種別を `isFormUpload` 引数で受け取る。

```csharp
// L866 付近（RDS/FS 分岐の直前）
if (ShouldScan(isFormUpload))
{
    for (var i = 0; i < files.Count; i++)
    {
        using var ms = new MemoryStream();
        files[i].InputStream.CopyTo(ms);
        var fileBytes = ms.ToArray();
        ms.Position = 0;

        var scanResult = ScanFileAsync(fileBytes, files[i].FileName)
            .GetAwaiter().GetResult();

        if (!scanResult.IsClean)
        {
            if (scanResult.IsError && !Parameters.VirusScan.RejectOnScanError)
            {
                // スキャンエラー時にアップロードを許可する設定
                continue;
            }
            // ウイルス検出 or スキャンエラー（拒否設定時）
            return scanResult.IsError
                ? Error.Types.VirusScanError
                    .MessageJson(context: context, scanResult.ErrorMessage)
                : Error.Types.VirusDetected
                    .MessageJson(context: context, scanResult.VirusName);
        }
    }
}
```

> **注意**: `InputStream` は一度読むとシークが必要。
> `MemoryStream` にコピーした後、**元のストリームを再セットする**か、
> コピー済みの `byte[]` を後続の保存処理に渡す方式を検討する必要がある。
> 実装時には `PostedFile` のストリーム管理を確認すること。

#### Step 3-4: API アップロードへのスキャン挿入

**ファイル**: `Implem.Pleasanter/Controllers/Api/BinariesController.cs`

**改修箇所**: `Upload` メソッド内、一時ファイル保存後・`ValidateFileHash` の後、
`ItemModel.UpdateByApi()` / `BinaryUtilities.CreateAttachment()` の前。

一時ファイルが `App_Data/Temp/{guid}/{fileName}` に保存済みのため、
ファイルパスから `byte[]` を読み出してスキャンする。

API コントローラーは `isFormUpload = false` として扱う。
`ScanMode == "All"` の場合のみスキャンが実行される。

```csharp
// 一時ファイル保存後
if (ShouldScan(isFormUpload: false))
{
    var fileBytes = System.IO.File.ReadAllBytes(tempFilePath);
    var scanResult = await virusScanService.ScanAsync(fileBytes, fileName);

    if (!scanResult.IsClean)
    {
        // 一時ファイルを削除
        System.IO.File.Delete(tempFilePath);

        return scanResult.IsError
            ? ApiResults.Error(context, new ErrorData(type: Error.Types.VirusScanError))
            : ApiResults.Error(context, new ErrorData(type: Error.Types.VirusDetected));
    }
}
```

## チャンクアップロード時の考慮事項

Pleasanter は `Content-Range` ヘッダーによるチャンクアップロードに対応している。
チャンクアップロード時のスキャンタイミングには以下の制約がある。

| 課題                   | 対応方針                                                                        |
| ---------------------- | ------------------------------------------------------------------------------- |
| 途中チャンクのスキャン | 不可。ファイル全体が揃ってからスキャンする必要がある                            |
| 完了判定               | `contentRange.To + 1 == contentRange.Length` で最終チャンクを判定               |
| スキャンタイミング     | 最終チャンク受信後、`ValidateFileHash` と同じタイミングでスキャン               |
| ファイル結合           | FS 保存時はファイルが Append されて完成済み。DB 時は `Bin` カラムに完全格納済み |

**実装方針**: チャンクアップロードの場合は、最終チャンク到着時
（`controlOnly == false` の条件、つまり完全なファイルが揃った時点）にのみスキャンを実行する。

```csharp
// contentRange == null（非チャンク）or 最終チャンク到着時のみスキャン
var isComplete = contentRange == null
    || (contentRange.To + 1 == contentRange.Length);

if (isComplete && ShouldScan(isFormUpload))
{
    // スキャン実行
}
```

## ScanMode 動作仕様

### 設定パターン

| VirusScan.json の状態    | `ScanMode` | 動作                                                 |
| ------------------------ | :--------: | ---------------------------------------------------- |
| ファイルが存在しない     |     -      | スキャン無効（`Read<T>(required: false)` で `null`） |
| `"ScanMode": "Disabled"` | `Disabled` | スキャン無効（`NullVirusScanService` 使用）          |
| `"ScanMode": "FormOnly"` | `FormOnly` | FormBinariesController 経由のみスキャン              |
| `"ScanMode": "All"`      |   `All`    | 全アップロード経路をスキャン                         |

### 無効時の動作保証

スキャン無効時は以下の動作を保証する。

1. `NullVirusScanService` が常に `VirusScanResult.Clean()` を返す
2. `ShouldScan()` のガード条件で早期リターン
3. nClam への TCP 接続は一切発生しない
4. パフォーマンスへの影響はゼロ（null チェック 1 回のみ）
5. ClamAV 未インストール環境でもエラーにならない

### 有効時のエラー制御

| 状況               | `RejectOnScanError` | 動作                         |
| ------------------ | :-----------------: | ---------------------------- |
| ウイルス検出       |        任意         | **常にアップロード拒否**     |
| clamd 接続失敗     |       `true`        | アップロード拒否             |
| clamd 接続失敗     |       `false`       | アップロード許可（ログ記録） |
| タイムアウト       |       `true`        | アップロード拒否             |
| タイムアウト       |       `false`       | アップロード許可（ログ記録） |
| ファイルサイズ超過 |        任意         | スキャンスキップ（許可）     |

## 実装上の注意事項

### CodeDefiner 自動生成との共存

| 注意点                   | 対応方針                                                       |
| ------------------------ | -------------------------------------------------------------- |
| `Error.cs` は自動生成    | 直接編集しない。Display JSON を追加して CodeDefiner を実行する |
| `Messages.cs` は自動生成 | 同上                                                           |
| CodeDefiner 実行順序     | Display JSON 配置 → CodeDefiner 実行 → ビルド確認              |
| バージョンアップ時       | Display JSON が残っていれば CodeDefiner 再実行で自動反映される |

### ストリーム管理

| 注意点                      | 対応方針                                                                                 |
| --------------------------- | ---------------------------------------------------------------------------------------- |
| `InputStream` は一方向      | スキャン前に `MemoryStream` にコピーし、コピーをスキャンに、元ストリームを後続処理に使う |
| DB 保存パスの二重読み取り   | スキャン→保存の順序でバイト配列を共有                                                    |
| FS 保存パスのファイル再読込 | 保存後のスキャンなら `File.ReadAllBytes()` で取得可能                                    |

### パフォーマンス影響

| 項目                 | 影響                                                       |
| -------------------- | ---------------------------------------------------------- |
| 無効時               | null チェック 1 回（ナノ秒オーダー） — 影響なし            |
| 有効時               | clamd へのラウンドトリップ + スキャン時間                  |
| 典型的なスキャン時間 | 小ファイル（< 1MB）: 50〜200ms / 大ファイル（10MB）: 1〜3s |
| 並行アップロード     | clamd は複数接続を並行処理可能                             |
| チャンクアップロード | 最終チャンクのみスキャン → 中間チャンクへの影響なし        |

## スキャン中のUI表示（アニメーション拡張）

### 現状のアップロードUI

Pleasanter のフロントエンドには 2 種類のローディング表示機構がある。

| 機構                    | CSS / JS                          | 用途                         |
| ----------------------- | --------------------------------- | ---------------------------- |
| `#LoaderContainer`      | `loading.js` / CSS rotate spinner | Ajax 通信中のフルスクリーン  |
| `.progress-bar`（個別） | `attachments.js` / jQuery animate | 添付ファイルごとのプログレス |

添付ファイルアップロードでは **`.progress-bar`（個別プログレスバー）** が使われている。
`#LoaderContainer` はレコード保存時等の Ajax で使用されるものであり、
ファイルアップロード中には表示されない。

### スキャン中の表示方針

アップロード完了後〜スキャン完了までの間にユーザーへフィードバックを返すため、
既存のプログレスバーを拡張する方式を推奨する。

#### 案A: プログレスバーの「スキャン中」状態拡張（推奨）

アップロード 100% 到達後、バーの表示を「スキャン中...」に切り替える。

```text
[████████████████████ 100% ] → [████████░░░░ スキャン中... ]
                                 ↑ 色を変えて脈動アニメーション
```

**実装ステップ**:

1. `attachments.js` の `done` コールバックでバーをスキャン表示に切替
2. CSS で `@keyframes pulse` アニメーション追加
3. サーバーサイドのスキャンレスポンスを受信後に完了/エラー表示

```javascript
// attachments.js の done コールバック内（概要）
done: function (e, data) {
    var $bar = statusbar.find('.progress-bar');
    $bar.css('width', '100%');
    // スキャン中状態に切替
    $bar.addClass('scanning');
    $bar.text($p.display('Scanning'));
    // サーバーレスポンスを待機（既にサーバーサイドでスキャン済み）
    // → レスポンス受信後に scanning クラスを除去
}
```

```css
/* スキャン中のパルスアニメーション */
.progress-bar.scanning {
    background-color: #f0ad4e; /* 警告色 */
    animation: pulse 1.5s ease-in-out infinite;
}

@keyframes pulse {
    0%,
    100% {
        opacity: 1;
    }
    50% {
        opacity: 0.6;
    }
}
```

#### 案B: `#LoaderContainer` を追加表示

スキャン中にフルスクリーンの `$p.loading()` を呼び出す簡易方式。

```javascript
// サーバーへの送信前にローダー表示
$p.loading();
// → サーバーサイドでアップロード + スキャンが完了
// → レスポンス受信後に $p.loaded() で非表示
```

**メリット**: 実装が非常に簡単（JS 1 行追加）
**デメリット**: 画面全体がブロックされるため UX が悪い。複数ファイルのどれがスキャン中か不明。

#### 案C: スキャン専用のステータスメッセージ

プログレスバーの下にテキストメッセージを表示する。

```html
<div class="scan-status">
    <span class="scan-spinner"></span>
    ウイルススキャン中...
</div>
```

### 推奨方式

| 方式 | 実装コスト | UX   | 推奨度 |
| ---- | :--------: | ---- | :----: |
| 案A  |     中     | 良好 |   ◎    |
| 案B  |     低     | 悪い |   △    |
| 案C  |     中     | 良好 |   ○    |

**案A（プログレスバー拡張）を推奨**。
ユーザーはアップロード→スキャンの遷移を直感的に理解でき、
ファイルごとの状態を把握できる。

### サーバーサイドとの連携

現在のアップロードフローでは、サーバーサイドでスキャンが完了してから
レスポンスが返る（同期的）ため、クライアント側でのポーリングは不要。

```mermaid
sequenceDiagram
    participant C as クライアント
    participant S as サーバー
    participant V as clamd

    C->>S: POST /binaries/upload
    Note left of C: プログレスバー表示
    S->>S: ファイル受信
    Note left of C: 100% → スキャン中表示
    S->>V: INSTREAM（スキャンリクエスト）
    V-->>S: スキャン結果
    S-->>C: JSON レスポンス
    Note left of C: 完了 or エラー表示
```

スキャンに時間がかかる場合（大ファイル）、クライアント側では
HTTP レスポンスのタイムアウトに注意する必要がある。
`TimeoutMs` パラメータとクライアント側の Ajax タイムアウトを一致させること。

## セキュリティ考慮事項（脆弱性分析）

ウイルススキャン機能の追加により新たに発生しうるセキュリティリスクと対処法を整理する。

### 脆弱性一覧

| #   | 脆弱性                               | 深刻度 | 攻撃シナリオ                                                                                | 対処法                                                                         |
| --- | ------------------------------------ | :----: | ------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| V1  | ClamAV 接続の中間者攻撃（MITM）      |   高   | clamd と Pleasanter 間の TCP 通信が暗号化されていないため、スキャン結果を改ざんされる可能性 | clamd を localhost または専用 VLAN に配置。Docker の場合は内部ネットワーク使用 |
| V2  | DoS（サービス拒否）攻撃              |   高   | 大量の大容量ファイルを連続アップロードし、clamd のリソースを枯渇させる                      | `MaxFileSizeMB` でサイズ制限、レートリミット導入、clamd の接続数上限設定       |
| V3  | スキャンバイパス（圧縮爆弾）         |   中   | zip bomb（圧縮展開で巨大化するファイル）を送信し、clamd のメモリ/CPU を枯渇させる           | clamd の `MaxScanSize`, `MaxFileSize`, `MaxRecursion` を設定                   |
| V4  | スキャンバイパス（分割アップロード） |   中   | マルウェアを複数チャンクに分割し、個別チャンクではパターンマッチを回避                      | 最終チャンク到着後に結合済みファイル全体をスキャン（実装済みの方針）           |
| V5  | タイムアウト悪用                     |   中   | スキャンタイムアウトを意図的に引き起こし、`RejectOnScanError: false` 時にスキャンを回避     | 本番環境では `RejectOnScanError: true` を推奨。タイムアウト時もログに記録      |
| V6  | パス・トラバーサル                   |   中   | 一時ファイル保存時のファイル名操作でディレクトリトラバーサルを試みる                        | `Path.GetFileName()` で正規化済み（既存処理で対応済み）                        |
| V7  | ClamAV 定義ファイルの陳腐化          |   中   | ウイルス定義が更新されないまま運用され、新型マルウェアを見逃す                              | freshclam の自動更新を有効化。更新状況の監視アラート設定                       |
| V8  | 誤った安心感（False Sense）          |   低   | 「スキャン済み」表示によりユーザーが過信し、別のセキュリティ対策を怠る                      | スキャンは多層防御の一要素であることをドキュメント・管理画面で明示             |
| V9  | スキャン結果のログ過剰出力           |   低   | ウイルス名やエラー詳細をログに過剰出力し、ログインジェクションやディスク圧迫に繋がる        | ログ出力内容をサニタイズ。ログローテーション設定                               |
| V10 | clamd のバージョン脆弱性             |   中   | clamd 自体に既知の脆弱性（CVE）が存在する場合、攻撃の踏み台になりうる                       | clamd を定期的にアップデート。セキュリティアドバイザリを監視                   |

### 対策チェックリスト

実装・デプロイ時に確認すべき項目:

- [ ] clamd を localhost または隔離ネットワーク内に配置しているか
- [ ] freshclam による定義ファイル自動更新が有効か
- [ ] `MaxScanSize` / `MaxFileSize` / `MaxRecursion` を設定しているか
- [ ] 本番環境で `RejectOnScanError: true` に設定しているか
- [ ] clamd のバージョンに既知の CVE がないか確認したか
- [ ] アップロードのレートリミットを検討したか
- [ ] スキャンエラー・タイムアウトが適切にログ記録されるか
- [ ] ログ出力内容にインジェクションリスクがないか

## 結論

| 項目             | 結論                                                                         |
| ---------------- | ---------------------------------------------------------------------------- |
| 改修規模         | 新規ファイル 8 件 + 既存改修 6 件 + CodeDefiner 自動生成 2 件                |
| スキャンモード   | `ScanMode`（3 パターン: All / FormOnly / Disabled）。未配置時は自動的に無効  |
| スキャンエンジン | ClamAV（clamd）+ nClam。DI で `IVirusScanService` として抽象化               |
| アップロード経路 | 3 つのコントローラー × 2 種のメソッド（ファイル / 画像）を網羅               |
| エラー追加方式   | CodeDefiner の Display JSON 方式で `Error.Types` / `Messages` を自動生成     |
| 主な課題         | ストリーム管理（一方向読み取り）とチャンクアップロード時のタイミング制御     |
| 将来の拡張       | `IVirusScanService` のインターフェースにより AMSI・VirusTotal 等への差替可能 |
