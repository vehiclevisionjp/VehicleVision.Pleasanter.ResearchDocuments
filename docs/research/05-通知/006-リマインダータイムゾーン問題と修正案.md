## 問題の概要

### 報告された症状

以下のような設定でリマインダーを作成した場合：

- **開始日時**: 2026/03/30 14:56
- **期間種別**: 毎日

**期待される動作**:

- 本日 14:56 にメール送信
- 翌日 14:56 に再度メール送信（24時間間隔）

**実際の動作**:

- **毎秒メールが送信される**
- Reminder.json の interval を 500 → 30000 に変更しても改善しない

### 根本原因

リマインダーのバックグラウンドタスクが**Service.json の TimeZoneDefault**を参照しているが、
データベースの DateTime 値は**TimeZoneInfo.Local（サーバーOS のタイムゾーン）**
で保存されているため、タイムゾーン不一致による計算誤りが発生している。

## タイムゾーンの3層構造

Pleasanter には3つのタイムゾーン設定が存在する：

| タイムゾーン           | 設定元                | 用途                                                        |
| ---------------------- | --------------------- | ----------------------------------------------------------- |
| **TimeZoneInfo.Local** | OS（変更不可）        | **DB保存時の基準TZ**（すべての DateTime 値）                |
| **TimeZoneDefault**    | Service.json          | システムデフォルト TZ（未認証時、バックグラウンドタスク用） |
| **User TimeZone**      | Users.TimeZone カラム | ユーザー個別の TZ（表示・入力用）                           |

### DateTime 値の保存・変換フロー

```
[DB保存]
User TZ（表示値）
    ↓ ToUniversal(context)
TimeZoneInfo.Local（サーバーOS TZ）← DB に保存
    ↓
[DB読み取り]
TimeZoneInfo.Local
    ↓ ToLocal(context)
User TZ（表示値）
```

**重要な仕様**:

- **すべての DateTime 値は TimeZoneInfo.Local で DB に保存される**（UTC ではない）
- `ToUniversal()` は "User TZ → Server Local TZ" への変換（UTC 変換ではない）
- `ToLocal()` は "Server Local TZ → User TZ" への変換

## リマインダーにおけるタイムゾーン不一致の詳細

### Context.TimeZoneInfo の初期化フロー

**通常のユーザーリクエスト**:

```csharp
// HTTPリクエスト時
var context = new Context(request: true);
// ↓ ログイン時
context.SetUser(userModel);
context.TimeZoneInfo = userModel.TimeZoneInfo;  // ユーザーの TZ
```

**バックグラウンドタスク（リマインダー含む）**:

```csharp
// バックグラウンドタスク時
var context = new Context(
    tenantId: tenantId,
    request: false,          // HTTPリクエストなし
    setAuthenticated: true
);
// context.TimeZoneInfo = Environments.TimeZoneInfoDefault
//                      ← Service.json の TimeZoneDefault
```

**問題点**:

- バックグラウンドタスクは `SetUser()` を呼ばない
- **Context.TimeZoneInfo が TimeZoneDefault のまま**
- DB 値は TimeZoneInfo.Local で保存されている
- **計算基準 TZ と DB 保存 TZ が不一致**

### 具体的な問題発生メカニズム

#### シナリオ: TimeZoneDefault ≠ TimeZoneInfo.Local

```
前提条件:
- Server Local TZ (TimeZoneInfo.Local) = JST (+9)
- Service.json TimeZoneDefault = UTC
- リマインダー設定: 毎日 14:56 JST

問題の流れ:

1. リマインダー作成時（ユーザーコンテキスト）
   ユーザー入力: 2026/03/30 14:56 JST
   → ToUniversal(JST context): 2026/03/30 14:56 JST
   → DB保存: 2026/03/30 14:56 JST（Server Local TZ）

2. バックグラウンド実行時（システムコンテキスト）
   context.TimeZoneInfo = UTC（TimeZoneDefault）

   DateTime.Now.ToLocal(context).Date
   = 現在時刻 JST → UTC 変換（-9時間）
   = 日付計算が9時間ずれる

3. ScheduledTime の比較
   DB値: 2026/03/30 14:56 JST
   比較基準: 計算時に再度 ToUniversal(UTC context) 適用
   → 二重変換により時刻が大幅にずれる
   → ScheduledTime ≤ DateTime.Now が常に true
   → 60秒ごとのチェックで毎回実行
```

#### Reminder.cs の問題コード箇所

**Reminder.cs (L565-623) - GetDataTable メソッド**:

```csharp
// 日付範囲の計算
DateTime.Now.ToLocal(context: context).Date  // ← TimeZoneDefault を使用
    .AddDays(range * -1)

// SQL WHERE 句に組み込み
where.Add(
    column: column,
    _operator: "<",
    value: endTime.ToString("yyyy/M/d H:m:s.fff")
);
```

**問題点**:

- `ToLocal(context)` が TimeZoneDefault を基準に計算
- DB 値は TimeZoneInfo.Local で保存されている
- 比較時に TZ オフセット分のズレが発生

**ReminderScheduleUtilities.cs - 実行判定**:

```csharp
// ScheduledTime との比較（推測）
var reminders = Rds.ExecuteTable(
    context: context,  // ← TimeZoneDefault のコンテキスト
    statements: Rds.SelectReminderSchedules()
        .Where(column: Rds.ReminderSchedulesColumn.ScheduledTime,
               value: context.Sqls.CurrentDateTime,
               _operator: "<=")
);
```

**問題点**:

- `context.Sqls.CurrentDateTime` は TimeZoneDefault 基準で評価される可能性
- DB の ScheduledTime は TimeZoneInfo.Local で保存されている
- 比較が正確に行われない

### interval パラメータについての誤解

**Reminder.json の interval パラメータは送信間隔を制御するものではない**。

実際の送信間隔は以下で決定される：

- **期間種別（Type）**: `Times.RepeatTypes` 列挙型（毎日、毎週、毎月など）
- **ScheduledTime**: `ReminderSchedules` テーブルに格納される次回実行時刻

ReminderBackgroundTimer は **60秒ごと**にチェックを行い、`ScheduledTime ≤ 現在時刻` の条件を満たすリマインダーを実行する。

## なぜ TimeZoneDefault を使っているのか

### 設計意図（部分的に意図的）

1. **テナントレベル操作としての設計**
    - リマインダーは「サイト設定」に基づく機能
    - 特定ユーザーではなくテナント全体の操作として設計された

2. **スケジュール一貫性の確保**
    - OS TZ（TimeZoneInfo.Local）はサーバー移行やクラスタ構成で変わる可能性がある
    - Service.json の TimeZoneDefault は明示的に設定され、環境に依存しない

3. **バックグラウンドタスク統一設計**
    - ReminderBackgroundTimer 以外のバックグラウンドタスクも同じパターン
    - DeleteSysLogsTimer、SessionGc など、すべて TimeZoneDefault を使用

### 設計ギャップ（非意図的な問題）

1. **マルチタイムゾーン環境への対応不足**
    - ユーザーが異なる TZ にいる場合、TimeZoneDefault では不十分
    - リマインダー作成者の TZ を保持・使用すべき

2. **DB保存 TZ との不整合**
    - **DB保存は TimeZoneInfo.Local（OS TZ）** ← これが正解
    - **計算は TimeZoneDefault** ← これが不一致の原因
    - 本来は**すべて TimeZoneInfo.Local で統一すべき**

## 正しい設計（修正案）

### なぜ TimeZoneInfo.Local を使うべきか

1. **DB保存と一貫性を保つ**

    ```
    DB保存：TimeZoneInfo.Local
    計算基準：TimeZoneInfo.Local  ← 同じTZで統一
    表示変換：User TimeZone
    ```

2. **DateTime.Now との直接比較が可能**

    ```csharp
    // 現在の実装（問題あり）
    DateTime.Now.ToLocal(context)  // TimeZoneDefault基準

    // 正しい実装
    DateTime.Now  // TimeZoneInfo.Local（DBと同じ基準）
    ```

3. **TZ変換の二重適用を回避**
    - 現状：DB値（Server Local TZ）→ ToUniversal（TimeZoneDefault 基準）→ 二重変換
    - 正解：DB値（Server Local TZ）→ そのまま比較（変換不要）

### 修正方針1: Context を TimeZoneInfo.Local で作成（推奨）

**ReminderBackgroundTimer.cs または ReminderScheduleUtilities.cs**:

```csharp
// 現在の実装
var context = new Context(
    tenantId: tenantId,
    request: false,
    setAuthenticated: true
);
// この時点で context.TimeZoneInfo = TimeZoneDefault

// 修正案
var context = new Context(
    tenantId: tenantId,
    request: false,
    setAuthenticated: true
);

// 明示的に Server Local TZ を設定
context.TimeZoneInfo = TimeZoneInfo.Local;  // ← この1行を追加

ReminderScheduleUtilities.Remind(context);
```

**メリット**:

- DB保存と計算基準が一致
- TZ変換が不要になり、パフォーマンス向上
- TimeZoneDefault との不一致問題が解消
- 既存のリマインダー設定への影響が最小限
- コード変更が最小（1〜2行追加）

**デメリット**:

- サーバー OS の TZ 設定に依存する
- マルチ TZ 環境では根本的な解決にならない

### 修正方針2: リマインダーに TimeZoneId を保持（より理想的）

**Reminder.cs - プロパティ追加**:

```csharp
public class Reminder : ISettingListItem
{
    public int Id { get; set; }
    public ReminderTypes ReminderType;
    public string Subject;
    public string Body;
    public string Line;
    public string From;
    public string To;
    public string Token;
    public string Column;
    public DateTime StartDateTime;
    public Times.RepeatTypes Type;
    public int Range;
    public bool? SendCompletedInPast;
    public bool? NotSendIfNotApplicable;
    public bool? NotSendHyperLink;
    public bool? ExcludeOverdue;
    public int Condition;
    public bool? Disabled;

    // 追加プロパティ
    public string ReminderTimeZoneId;  // リマインダー作成者のTZ
}
```

**実行時の使用**:

```csharp
// ReminderScheduleUtilities.cs
var reminderTz = TimeZoneInfo.FindSystemTimeZoneById(
    reminder.ReminderTimeZoneId ?? TimeZoneInfo.Local.Id
);

var context = new Context(
    tenantId: tenantId,
    request: false,
    setAuthenticated: true
);

// リマインダー固有の TZ を設定
context.TimeZoneInfo = reminderTz;
```

**メリット**:

- マルチ TZ 環境で正確な動作
- リマインダーごとに異なる TZ 指定可能
- ユーザーの TZ 変更に影響されない
- 将来的な拡張性が高い

**デメリット**:

- データベーススキーマ変更が必要
- 既存リマインダーのマイグレーション必要
- UI の変更（TZ 選択機能追加）が必要
- 実装工数が大きい

## Service.json TimeZoneDefault と TimeZoneInfo.Local の不一致による影響

### 影響範囲

TimeZoneDefault が実行環境の実際のタイムゾーン（TimeZoneInfo.Local）と食い違うと、以下の問題が発生する：

| 機能                   | 影響度 | 発生条件                    | 具体的な問題                         |
| ---------------------- | ------ | --------------------------- | ------------------------------------ |
| **リマインダー送信**   | ⚠️ 高  | TimeZoneDefault ≠ Server TZ | 間違った時刻に送信、または連続送信   |
| 期限切れ判定           | ⚠️ 高  | User TZ ≠ Server TZ         | 判定が遅延                           |
| カレンダー表示         | ⚠️ 中  | Ymd + User TZ ≠ Server TZ   | 終日イベントが時間付きイベントになる |
| 日付フィールド         | ⚠️ 高  | Ymd + TZ境界越え            | 日付が1日ずれる                      |
| バックグラウンドタスク | ⚠️ 高  | TimeZoneDefault ≠ Server TZ | 予定外の時刻に実行                   |
| CSV エクスポート       | ⚠️ 中  | マルチ TZ 環境              | ユーザーごとに異なる日付が出力       |
| 日付フィルター         | ⚠️ 中  | User TZ ≠ Server TZ         | 誤ったレコードが選択される           |
| ServerScript           | ⚠️ 中  | Server TZ ≠ UTC             | 日付が二重シフトする                 |

### 推奨される運用設定

1. **TimeZoneDefault は必ず実サーバーの OS TZ と一致させる**

    ```json
    // Service.json
    {
        "TimeZoneDefault": "Tokyo Standard Time" // OS TZ と同じ
    }
    ```

2. **サーバーの OS TZ を途中で変更しない**
    - 既存の DB 値との整合性が完全に崩れる

3. **マルチ TZ 環境では UTC を推奨**

    ```json
    // Service.json
    {
        "TimeZoneDefault": "UTC"
    }
    ```

    - サーバー OS も UTC に設定
    - すべてのユーザーは Users.TimeZone で個別設定

## 結論

### 問題の本質

リマインダーが毎秒送信される問題は、以下の3つの要因による：

1. **TimeZone 二重変換問題**: リマインダー作成時と実行時で異なる TZ コンテキスト
2. **ScheduledTime 更新の失敗**: 次回実行時刻が正しく進まない
3. **Server Local TZ と System Default TZ の不一致**: Service.json の設定がサーバー OS と異なる

### 推奨される修正

**短期対応（推奨）**:

```csharp
// ReminderBackgroundTimer または ReminderScheduleUtilities
var context = new Context(tenantId: tenantId, request: false, setAuthenticated: true);
context.TimeZoneInfo = TimeZoneInfo.Local;  // この1行を追加
```

**長期対応（理想）**:

- Reminder クラスに ReminderTimeZoneId プロパティを追加
- リマインダーごとに TZ を保持・使用
- マルチ TZ 環境への完全対応

### 運用上の推奨事項

1. **Service.json の TimeZoneDefault を実サーバー OS TZ と一致させる**
2. **サーバー OS の TZ を途中で変更しない**
3. **マルチ TZ 環境では UTC を推奨**
