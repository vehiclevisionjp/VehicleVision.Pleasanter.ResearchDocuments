## 調査目的

- DB 周りとセッション周りの実装を再点検し、パフォーマンス劣化を招き得る箇所を洗い出す
- 根本原因を伴った改善案を提示し、具体的な改修着手ポイントを示す

## 観測したボトルネック

### セッション削除が限定的でフルスキャンになりやすい

- `DeleteOldSessions` は日付変化時の 1 リクエスト目で `UpdatedTime < now - RetentionPeriod` を満たす行を DELETE。
  `SessionGuid not like '@%'` でユーザー保存セッションを対象外としており、`@userId` のデータが蓄積し続ける。`Implem.Pleasanter/Models/Sessions/SessionUtilities.cs:434-448`
- `App_Data/Definitions` 配下に Sessions 用の補助インデックス定義が見当たらず（主キーは `SessionGuid`+`Key`+`Page` のみ）、`UpdatedTime` 条件はフルスキャンになりがち。
- RetentionPeriod（分）を超える行が大量にある場合、朝イチのリクエストで DELETE が長時間ロックを握り、他のクエリをブロックし得る。

### Redis セッション読取が全フィールド取得

- Redis 使用時も `HashGetAll` でセッションハッシュ全体を取得し、`Name.Split('_')` でページ判定してフィルタリング。ページ数×キー数に比例して無駄な転送・文字列 Split が発生。`Implem.Pleasanter/Models/Sessions/SessionUtilities.cs:33-52`
- `readOnce` についても `_readOnce` サフィックスのキーを別途 `KeyExists` → `HashGetAll` で再取得しており、リクエストあたり 2 往復。
- ページ単位に記録される `View`/`Message` 等が多い環境では、Redis 帯域と CPU がボトルネックになる。

### テナント情報の毎リクエスト読込

- HTTP リクエストごとに `SetTenantProperties` が実行され、`Tenants` から 14 カラム（契約設定 JSON、TopStyle/TopScript など重量級）を全件読み出す。`Implem.Pleasanter/Libraries/Requests/Context.cs:493-535`
- 変更頻度が低い設定値にもかかわらずキャッシュなしで都度取得しており、高負荷時には Tenants テーブルへの不要な I/O が蓄積。

### 更新監視のためのステータス問い合わせが毎リクエスト実行

- 認可フィルターから毎リクエスト `SiteInfo.Refresh` が呼ばれ、`UpdateMonitor.Monitor` が `Statuses` テーブルを SELECT して更新時刻を確認する。
  `Implem.Pleasanter/Filters/CheckContextAttributes.cs:115`, `Implem.Pleasanter/Libraries/Server/SiteInfo.cs:23-119`
- 更新がない場合でも監視クエリが必ず走るため、高リクエスト環境では監視目的の I/O が無視できないコストになる。

## 改善案

1. **セッション削除の最適化**
    - `UpdatedTime` + `SessionGuid` で複合インデックスを追加し、日次クリーンアップのスキャン量を削減。
    - ユーザー保存セッション（`@userId`）にも TTL を設定し、`DeleteOldSessions` の WHERE から除外しない。
    - 日次バッチ/バックグラウンドジョブで一定件数ずつ削除する方式に変更し、リクエスト応答経路から切り離す。
2. **Redis セッションのページ単位取得**
    - フィールドを `key` と `key:{page}` に分けるのではなく、ページごとにハッシュキーを分割するか、`HashScan` のパターン指定でページ一致のみ取得する。
    - read-once 用の削除は `HashDelete` とパイプラインでまとめ、`KeyExists` → `HashGetAll` の二重往復を解消。
3. **テナント設定のキャッシュ**
    - `Tenants` 行を `TenantId` 単位でメモリキャッシュし、`Statuses` の更新時刻か `Tenants.UpdatedTime` を監視して無効化する。
    - 必要なカラムだけを選択する軽量クエリと、スタイル/スクリプト等重量カラムを遅延読み込みする 2 段階構成を検討。
4. **更新監視のクエリ削減**
    - `Statuses` の更新時刻を短時間（例: 30〜60 秒）メモリキャッシュし、同期間のリクエストでは DB への確認をスキップ。
    - 代替案として、バックグラウンドで `Statuses` をポーリングし、差分があればキャッシュを更新する方式に変更。
